# OmniMind 변경사항 커밋

이 저장소의 절대 규칙: **`npm run verify`(lint → typecheck → test → build)를 통과하지 못한 커밋은 만들지 않는다.**
main은 Vercel 배포 라인이고 CI가 push마다 같은 검증을 돌리므로, 깨진 커밋은 배포 실패와 CI 빨간불로 즉시 되돌아온다.

## 절차

1. **변경 내용 파악** — `git status`와 `git diff`(+ `git diff --staged`)로 무엇이 바뀌었는지 실제로 읽는다.
   커밋 메시지는 diff를 읽어야 쓸 수 있다. 파일 목록만 보고 추측하지 않는다.
   - 의도치 않은 파일(비밀키, `.env*`, 대용량 산출물, 디버그 잔여물)이 섞여 있으면 커밋에서 제외하고 사용자에게 알린다.

2. **검증** — `npm run verify` 실행. build까지 포함해 몇 분 걸릴 수 있으니 타임아웃을 10분으로 잡는다.
   - 실패하면 **커밋하지 않는다.** 실패 원인이 방금 변경분의 사소한 문제(lint, 타입)면 고치고 재검증한다.
     원인이 불명확하거나 수정이 별도 작업 규모면 멈추고 실패 로그와 함께 사용자에게 보고한다.
   - 검증을 건너뛴 채 커밋해달라는 명시적 지시가 있을 때만 예외로 하되, 커밋 후 보고에 검증 미실시를 명시한다.

3. **논리 단위로 나누기** — 변경이 서로 다른 관심사(예: 엔진 수정 + 문서 갱신)를 담고 있으면 커밋을 나눈다.
   히스토리가 곧 이 프로젝트의 작업 로그다(로드맵 커밋들이 그 예). 단일 관심사면 한 커밋이면 된다.
   - verify는 **최종 트리 기준**이다. 커밋을 나누면 중간 커밋 시점의 트리는 개별 검증되지 않으므로,
     코드 커밋을 먼저·문서 커밋을 나중에 넣어 중간 커밋이 깨진 코드 상태가 되지 않게 한다.

4. **커밋** — `git add`(파일 명시, `-A` 남발 금지) 후 커밋.

## 커밋 메시지 규칙

이 저장소의 실제 히스토리를 따른다: **`type(scope): 한국어 요약`**, 세부 항목은 `·`(가운뎃점)으로 연결.

```
feat(daeun): 대운 마무리 — 계절 해석 연결·성별 컬럼 방어 저장·톤가드 경고 레벨
fix(auth): 콜백 실패 사유를 로그인 화면까지 전달 — Google 로그인 진단
docs(roadmap): 2026-07-15 현행화 — 마이그레이션 0001~0006 완료
```

- type: `feat` / `fix` / `docs` / `chore` / `refactor` / `test`
- scope: 기능 영역(auth, engine, daily, match, pro, roadmap …). 애매하면 최상위 디렉터리나 기능명.
- 요약: 한국어, "무엇을 왜"가 드러나게. 부연이 필요하면 `—` 뒤에 핵심 항목을 `·`로 나열.

## 푸시

커밋 자체는 로컬 작업이다. 사용자가 push까지 요청했거나 흐름상 명백하면 `git push`(main이면 `origin main`,
기능 브랜치면 `-u origin <branch>`)까지 하고, 아니면 커밋 완료 보고에 "push는 안 했다"고 명시한다.

## 완료 보고

만든 커밋(해시 + 메시지), verify 결과, push 여부를 사용자에게 보고한다.
