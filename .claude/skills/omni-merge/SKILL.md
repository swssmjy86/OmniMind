# OmniMind 코드리뷰 후 머지

기능 브랜치의 작업을 main에 합치기 전에 **리뷰 → 검증 → 머지**를 한 흐름으로 처리한다.
main은 곧 배포이므로, 이 스킬의 존재 이유는 "리뷰 안 된 코드가 main에 들어가는 일"을 없애는 것이다.

## 절차

### 1. 사전 상태 확인

- `git branch --show-current` — 기능 브랜치 위인지 확인한다. main 위라면 머지할 대상이 없으므로 상황을 파악해 사용자에게 보고한다.
- `git status` — 미커밋 변경이 있으면 먼저 `omni-commit` 스킬 절차대로 커밋한다(verify 포함). 리뷰는 커밋된 상태를 대상으로 해야 재현 가능하다.
- `git log main..HEAD --oneline` — 머지될 커밋 범위를 파악한다. 비어 있으면 머지할 것이 없다.

### 2. 코드리뷰

`/code-review` 스킬이 사용 가능하면 그것으로 현재 브랜치의 diff(main 대비)를 리뷰한다.
사용할 수 없으면 직접 리뷰한다: `git diff main...HEAD`를 실제로 읽고 다음을 본다.

- **정합성 버그** — 잘못된 조건, 누락된 예외 처리, 경계값. 특히 `src/lib/engine/`은 순수 계산 계층이므로 네트워크·IO·`Date.now()`·로컬 `Date` 게터가 섞여 들어왔는지 확인한다(이 저장소의 명시적 금지 사항).
- **프로젝트 규칙 위반** — 계산 로직에 LLM 호출이 섞임, 엔진 변경에 테스트 없음(TDD 필수 영역), 사용자 대면 문구가 브랜딩 톤(공감형 문체)에서 벗어남.
- **보안·데이터** — 키 노출, Supabase 쿼리의 권한 누락.

리뷰 결과 처리:
- **머지를 막아야 하는 결함**(버그, 규칙 위반)이 있으면 → 고치고 커밋한 뒤 재리뷰한다. 자신 없는 수정이면 멈추고 사용자에게 발견 사항을 보고한다. 결함을 안고 머지하지 않는다.
- **사소한 개선 제안**만 있으면 → 머지를 막지 말고, 완료 보고에 제안 목록으로 남긴다.

### 3. 검증

`npm run verify` (타임아웃 10분). 실패하면 머지하지 않는다 — 원인을 고치거나 사용자에게 보고한다.

### 4. 머지

```
git switch main
git pull --ff-only origin main        # main 최신화; 실패 시 멈추고 보고
git merge --no-ff <기능브랜치> -m "merge: <브랜치명> — <기능 한 줄 요약>"
```

`--no-ff`를 쓰는 이유: 기능 단위가 히스토리에 머지 커밋으로 남아, 나중에 기능 전체를 되돌리거나 추적하기 쉽다.

- **충돌 발생 시**: 기계적으로 풀리는 충돌(양쪽이 다른 파일/다른 영역)은 해소하고 진행한다.
  의미가 겹치는 충돌(같은 로직을 양쪽이 다르게 수정)은 임의로 풀지 말고 머지를 중단(`git merge --abort`)하고 사용자에게 보고한다.
- **머지 후, push 전에 main에서 `npm run verify`를 한 번 더 돌린다.** 브랜치와 main이 각자 verify를 통과해도
  합치면 깨지는 의미 충돌(한쪽이 바꾼 API를 다른 쪽 새 코드가 옛 방식으로 호출하는 류)은 머지된 트리에서만 드러난다.
  실패하면 push하지 않는다 — 원인이 명확하면 수정 커밋을 얹고, 아니면 사용자에게 보고한다.

### 5. 마무리

```
git push origin main
git branch -d <기능브랜치>            # 원격에도 있으면 git push origin --delete <기능브랜치>
```

push 실패(권한, 네트워크)는 머지 자체를 되돌리지 말고 상태 그대로 보고한다.

## 완료 보고

사용자에게: 리뷰 결과(발견·수정한 사항, 남긴 제안), verify 결과, 머지 커밋 해시, push 여부, 브랜치 정리 여부.
